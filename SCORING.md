# SCORING — требования

Подход: **rubric-judged, model-routed**. Никаких ручных сумм весов.
Конвейер: Haiku extract → ленивый детерминированный prefilter →
Sonnet rubric score (с полем «Обоснование») → детерминированный salary-floor guard.

## Шкала
- relevance_score: 0–100 (100 = идеальное совпадение). Целое, кламп в [0,100].
- Порог surface T = 60. Применяется детерминированно к LLM-скору:
  score ≥ T → surfaced, иначе → rejected.

## Маршрутизация моделей
- **EXTRACTION (массово, дёшево):** `claude-haiku-4-5` извлекает каждый пост в
  схему §4 (это существующий extract-шаг; остаётся на Haiku).
- **SCORE (суждение):** `claude-sonnet-4-6` оценивает РОЛЬ/обязанности по рубрике
  ниже и возвращает `relevance_score` + человекочитаемое «Обоснование».
- Оба id настраиваются через env (`ANTHROPIC_CHEAP_MODEL`, `ANTHROPIC_JUDGE_MODEL`),
  Haiku — дефолт для дешёвых шагов.

## 1) Ленивый prefilter (детерминированный, pure)
Отсекает ТОЛЬКО очевидно-не-вакансии:
- пустой/мусорный/слишком короткий текст (стикеры, пинги),
- посты-«ищу работу» / #резюме / open to work.

Высокий recall: **когда сомневаешься — оставляй**. Это НЕ скоринг релевантности.
Выжившие идут к Sonnet. (`scoring.prefilter`, чистая функция.)

## 2) Sonnet rubric score (поле «Обоснование»)
Sonnet судит по СТЕКУ и обязанностям, НЕ по тайтлу (вакансии называются по-разному).
Текст «Обоснование» — это surface прозрачности (показывается на карточке).
Рубрика (judge-инструкции + структура STRONG FIT / LOCATION PRIORITY / FLAG /
LOW-REJECT + формат вывода) живёт в коде — каркас `_SCORE_SCAFFOLD` в
`job_hunter/llm.py`. ПРОФИЛЬ кандидата (персональные данные) НЕ хардкодится:
он грузится в рантайме из YAML (`config/profile.local.yaml`, иначе
`config/profile.example.yaml`) лоадером `job_hunter/profile.py` и
вставляется в системный промпт через `llm.build_score_system(profile)`
(блок `render_profile_block`). Публичный репозиторий содержит только
ОБОБЩЁННЫЙ пример профиля — реальные данные лежат в gitignored
`profile.local.yaml`.

**Формат «Обоснование» (только презентация, рубрику НЕ меняет):** сканируемый
вердикт + буллеты на языке поста:
- **Строка 1** — ВЕРДИКТ, ВКЛЮЧАЮЩИЙ число скора (напр. «Сильный фит по роли —
  82/100» / «Strong role fit — 82/100»).
- **Затем 2–4 БУЛЛЕТА**, каждый начинается с ✅ (плюс / fit) или ⚠️ (риск /
  оговорка). Покрывают ОСНОВНОЙ fit (роль / стек / грейд: владеет ли она
  поведением модели, есть ли пересечение по стеку, верный ли грейд) И ГЛАВНУЮ
  оговорку / риск.
- **⚠️ ЗАРЕЗЕРВИРОВАН** для настоящих HARD-промахов / реальных рисков (см.
  REQUIREMENT SEVERITY ниже). Пробел по SOFT-требованию («будет плюсом» /
  «preferred»), которого нет у кандидата, — это НЕ ⚠️ «риск отказа»: упоминаем
  его в более мягкой, не-⚠️ формулировке (нейтральная оговорка или ✅-с-нюансом).
- **КРИТИЧНО:** каждый буллет — ПОЛНОЕ короткое ПРЕДЛОЖЕНИЕ, содержательное и
  самодостаточное, а НЕ рубленый фрагмент и НЕ одно слово.
  ХОРОШО: «✅ Роль — про внедрение LLM (RAG, prompt-eng), что точно совпадает с
  её хендз-он опытом.» ПЛОХО (фрагмент, ЗАПРЕЩЕНО): «✅ RAG».
- Держим сканируемость (вердикт + буллеты), но каждый буллет — целое
  предложение. НЕ простыня-абзац, НЕ однословные фрагменты.

JSON-контракт без изменений: `{"relevance_score": int 0–100, "Обоснование": "<строка-вердикт со скором + 2–4 ✅/⚠️ буллета-предложения>"}`.

### Профиль кандидата (CANDIDATE PROFILE) — грузится из YAML

Персональный профиль НЕ хранится в репозитории. Он загружается из
`config/profile.local.yaml` (реальный, gitignored) либо, если его нет, из
`config/profile.example.yaml` (обобщённый пример, закоммичен). Поля профиля
(`role`, `target_grade`, `hands_on`, `in_development`, `stack`, `languages`,
`location_priority`, `gender`, `salary_floor_eur`, `draft_signature`)
рендерятся в блок CANDIDATE PROFILE системного промпта функцией
`render_profile_block` и инжектятся в `_SCORE_SCAFFOLD`.

Полный список полей и пример значений — в `config/profile.example.yaml`.
Salary-floor (€/мес) тоже берётся из профиля (`salary_floor_eur`), а не
зашит в код.

### Рубрика (каркас в коде, профиль — данные)
- **STRONG FIT (высокий):** роли, где кандидат ВЛАДЕЕТ AI / поведением модели —
  Prompt/AI/LLM Engineer, AI Product Engineer, applied-LLM; где RAG/роутинг/
  промпты — ЯДРО работы; грейд совпадает с `target_grade` из профиля.
- **LOCATION PRIORITY:** локация оценивается по лестнице `location_priority`
  из профиля (релокация / remote / гибрид / офис — в порядке предпочтения).
  Роли за рубежом с релокацией / визовым спонсорством ранжируются высоко,
  если релокация заявлена как цель в профиле.
- **FLAG / DOWN-WEIGHT (НЕ reject, просто ниже скор):** зарплата не указана;
  роль, требующая свободного английского «прямо сейчас», если в профиле это
  отмечено как текущий пробел; senior-роль, требующая доказанного стажа /
  формальной записи о трудоустройстве.
- **LOW / REJECT-WORTHY (низкий скор):** backend-разработчик с LLM «сбоку»
  (underfit); чистый PM; маркетинг / lead-gen; DS / обучение ML-моделей;
  Python-лайвкодинг с нуля / хардкорное алго-интервью (почти гарантированный
  провал); HARD (с формулировкой «обязательно») требование профильного высшего
  образования, когда в профиле указано, что его нет → flag + низкий скор (это
  применяется ТОЛЬКО когда пост формулирует диплом как HARD — см. REQUIREMENT
  SEVERITY ниже; диплом, который лишь «будет плюсом / preferred», — это SOFT и
  даёт только лёгкое снижение, БЕЗ флага); лид / менеджмент → не fit.

### REQUIREMENT SEVERITY (HARD vs SOFT) — каркас в коде
Модель классифицирует КАЖДОЕ требование по СОБСТВЕННОЙ ФОРМУЛИРОВКЕ поста:
- **HARD** = пост формулирует как обязательное: «обязательно» / «required» /
  «must» / «необходимо» / «строго» / «mandatory» (и эквиваленты).
- **SOFT** = пост формулирует как опциональное / желательное: «будет плюсом» /
  «желательно» / «preferred» / «nice to have» / «as a plus» / «приветствуется»
  (и эквиваленты).
- SOFT-требование, которого нет у кандидата → ТОЛЬКО небольшое снижение скора;
  это НЕ ⚠️ hard-флаг и НЕ формулируется как «риск отказа» / блокер — отмечаем
  более мягким, нейтральным языком (мягкая оговорка или ✅-с-нюансом).
- HARD-требование, которого нет у кандидата → прежнее поведение: flag, ⚠️
  допустим, может снижать скор как сегодня.
- НЕОДНОЗНАЧНАЯ формулировка (пост явно НЕ говорит ни обязательно, ни
  опционально) → ПО УМОЛЧАНИЮ SOFT. Не выдумываем hard-серьёзность.

Парсер строгий: `relevance_score` (int 0–100, кламп) + строка «Обоснование».
Модель НЕ учитывает порог зарплаты — этим занимается детерминированный guard (§3).
(`llm.build_score_prompt` / `llm.parse_score_response`, вызов в `llm.llm_score`,
оркестрация в `pipeline._do_score`.)

## 3) Hard salary guard (детерминированный, перекрывает LLM)
Порог — **€N / месяц gross** (эквивалент), где N = `salary_floor_eur` из
загруженного профиля (`config/profile.*.yaml`); в коде НЕ зашит. Независимо от
LLM-скора:
- Зарплата указана И верх вилки < эквивалента €N/мес → reject
  (reason: `salary below EUR N/mo floor`).
- Зарплата НЕ указана → НЕ reject (поведение flag + penalty не меняется).

Реализация через существующий FX (`fx.py`): и верх вилки, и €N/мес
конвертируются в ОДНУ валюту (₽) живым курсом, после чего сравниваются — поэтому
порог работает для любой валюты вакансии. Чистая функция
`scoring.salary_guard_reject(salary_max_rub, floor_rub)` только сравнивает два
уже сконвертированных ₽-числа; сама FX-конверсия живёт в pipeline
(`_salary_max_rub` для вилки и `_salary_floor_rub` для €N, который берётся из
`deps.profile.salary_floor_eur`) и применяется на T3/T4 независимо от модели.
Если FX недоступен (floor=None) — guard не срабатывает.

Допущение: поля `salary_min/max` трактуются как МЕСЯЧНЫЕ (как и для ₽-карточки),
поэтому €N сравнивается как помесячная сумма.

## Валюта
- Хранить оригинал: salary_min/max + currency.
- Рядом показывать ₽-эквивалент, напр. «€3000 (~290k ₽)» — оригинал НЕ заменяется.
- Курс живой из бесплатного источника без ключа (frankfurter.app или open.er-api.com),
  кэш на сутки. Reject-проверка идёт по ₽-эквиваленту.

## contact_type
- enum dm | form | link, неизвестное → null.
- `contact` — контакт рекрутёра ИЗ ТЕЛА вакансии (@handle / email / ссылка-отклик),
  НЕ канал-источник. Канал/пермалинк — в source_channel / source_link.
